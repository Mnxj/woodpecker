### Why do frontend monitoring?

Find and resolve issues faster, drive product decisions, and enable new business possibilities.

### How to build a solid frontend monitoring strategy?

- **Metrics**:
   - Performance: FP, FCP, LCP, FID, CLS, etc.
   - Errors: JS errors, resource load errors, API errors.
   - User behavior: PV, UV, click paths, dwell time.
- **Implementation**:
   - Open-source tools: Sentry, LogRocket, Google Analytics.
   - Custom: collect data via `PerformanceObserver`, `Error` events, etc.
   - Reporting: use `navigator.sendBeacon` or `XMLHttpRequest`.

### Frontend monitoring goals

Stability

- JS errors: runtime errors, promise rejections
- Resource errors: JS/CSS load failures
- API errors: AJAX/fetch failures
- Blank screen: empty page

User experience

- PV: pageviews and clicks
- UV: distinct IPs visiting
- Dwell time per page

### Common instrumentation strategies

Code instrumentation

Insert tracking code directly.

Pros: precise (any moment, full data).

Cons: a lot of code work.

Visual instrumentation

Use a system to replace manual code insertion.

Auto instrumentation

Collect everything — no missed or wrong tracking.

Cons: more pressure on data transport and servers; can't easily customize data structure.

### Performance metrics

Common frontend perf metrics and how to compute them:

1. First Contentful Paint (FCP) — time to first content render
   - Computed: the browser records a timestamp when content is first painted to the screen.

2. Speed Index — visual completion speed
   - Measures the visual fill speed of a page.
   - Computed: analyzes captured video of load to compute the time it takes for content to reach a visual completion level.

3. Time to Interactive (TTI) — time to reliably interactive
   - When the main content is loaded and the main thread is idle for a while.

4. First Input Delay (FID) — first input latency
   - Measures response delay for the first user interaction (clicks, etc.).
   - Computed: difference between timestamp of first input and when the browser actually starts handling it.

5. Total Blocking Time (TBT) — total blocking time
   - Computed: from FCP to TTI, the sum of time intervals where the main thread is blocked > 50ms.

6. Largest Contentful Paint (LCP) — time of largest content paint
   - Computed: timestamp when the largest visible content element is rendered.

7. Cumulative Layout Shift (CLS) — cumulative layout shifts
   - Sum of all unexpected layout shifts during page load.

8. Resource Timing — resource load timing
   - Via browser's Resource Timing API.

9. Server Response Time
   - Difference between request-sent timestamp and start-of-response timestamp.

10. HTTP Requests
   - Count of requests in the DevTools network panel.



The `window.performance` API gives detailed load timing — `performance.now()` for intervals, `performance.getEntriesByType('resource')` for resource timing, etc.

`PerformanceObserver` API lets you observe perf events like resource loads and marks — register one to capture `largest-contentful-paint`, `first-input-delay`, etc.



### How to get a unique browser identifier?

- Use the fingerprinting library `fingerprintjs2`

  ```js
  const fpPromise = Fingerprint2.get({ extraPlugins: true });

  fpPromise.then(output => {
    console.log(output);
  });
  ```

- Canvas fingerprint (may be limited)

  ```js
  // It draws on a Canvas and reads pixel data to create a fingerprint.
  // Different browsers/devices render Canvas slightly differently, useful as part of a fingerprint.
  function getCanvasFingerprint() {
    const canvas = document.createElement('canvas');
     const context = canvas.getContext('2d');
     context.font = '14px Arial';
     context.fillText('Hello, world!', 2, 10);
     return canvas.toDataURL();
  }
  ```

### How does a server-side app get the client IP

`X-Forwarded-For` — usually added by proxies (load balancers, reverse proxies) to record the original client IP

// const clientIp = req.headers['x-forwarded-for'] || req.connection.remoteAddress;

`req.connection.remoteAddress` / `req.socket.remoteAddress` typically represent the IP of the client directly connected to the server.

#### `navigator.sendBeacon`

`navigator.sendBeacon` is a Web API that asynchronously sends data to a server, often used to send small amounts of data during page unload (e.g. when the user closes the tab or navigates away). It solves the issue of AJAX requests potentially being dropped on unload.

### Key features

1. **Non-blocking**: doesn't block page unload — the browser can still send data successfully.
2. **Supported formats**: simple text (strings) or `Blob`; typically used for JSON or form data.
3. **Use cases**:
   - Send analytics data (pageviews, event tracking).
   - Collect user behavior.
   - Send logs or error reports.

### Reporting methods? ajax, img, navigator.sendBeacon

Use `navigator.sendBeacon` to report data:

```javascript
navigator.sendBeacon('/log', JSON.stringify({ event: 'page_view' }));
```

### Ideal FCP and influencing factors

FCP's ideal value is < 1 second. Factors:

- Resource load time
- Server response time
- Browser parse and render time

### What's "good" vs "bad" for perf metrics?

Standards are usually based on user experience — shorter response time is better.

### Server response time impact on FCP and SI

**User experience**: longer server response means later FCP and SI, hurting initial UX. Longer time-to-content drops satisfaction and retention.

**SEO impact**: search engines increasingly weight page load perf; slow FCP/SI may hurt rankings.

**Optimization suggestions**:

- **Server**: improve response speed; use caching, CDN to reduce latency.
- **Assets**: optimize static assets, lazy loading, compression.
- **Monitoring**: regularly analyze server response times and UX metrics.
