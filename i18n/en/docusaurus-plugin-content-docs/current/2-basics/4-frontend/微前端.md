

> Splits a large frontend app into multiple modules. Each frontend module can be owned by a different team, choose its own framework, and be deployed independently.
>
> Common in: admin/back-office systems

### Why upgrade to microservices?

50k+: monolithic architecture

100k–millions: monolithic deployed across multiple servers (Nginx load balancing)

Millions to tens of millions: load balancing + master/slave database (heavier ops)

Microservices: split the monolith into multiple projects deployed across multiple servers

### Benefits

- Tech-stack agnostic
  The host framework doesn't restrict sub-app tech stacks; sub-apps are fully autonomous.

- Independent development, runtime, and deployment
  Sub-app repos are independent; FE and BE can be developed separately; after deployment, the host syncs automatically.

- Incremental upgrades

  For complex existing systems, full-stack upgrades or rewrites are rarely feasible — micro-frontends are an excellent strategy for incremental refactoring.

### How micro-frontends work

1. **Independent deployment**: each micro-frontend can be built, tested, and deployed independently.
2. **Independent runtime**: micro-frontends are independent at runtime; they can load in parallel without interfering with each other.
3. **Communication mechanisms**: micro-frontends need communication mechanisms to coordinate behavior — via pub/sub, event bus, Web Components, etc.
4. **Shared dependencies**: to reduce duplication and improve efficiency, micro-frontends can share common deps like styling libraries, utility libraries, etc.
5. **Route management**: a mechanism to manage routes so users see the right micro-frontend as they switch.

### Monorepo

1. **Code sharing**: projects/modules in the same repo can easily share code and resources.
2. **Unified dependency management**: all projects/modules share a single `package.json`, making deps simpler and consistent.
3. **Unified build/test**: a single build/test pipeline ensures quality and consistency across projects.
4. **Version control**: since everything's in one repo, version control is simpler and more intuitive.

### Combining micro-frontends with monorepos

- Putting all micro-frontends in the same repo simplifies dependency management and version control, and makes sharing common code easy.
- Helps maintain a unified build/test pipeline for quality and consistency.

### Frameworks

1. Iframe:
    - Downsides
        - URLs don't sync; refresh loses the iframe's route
        - Global contexts are fully isolated; memory isn't shared
        - UI doesn't sync
        - Slow — every sub-app entry reloads resources
2. **Single-SPA**:
    - Supports multiple frontend frameworks (React, Vue.js, Angular, etc.)
    - Advantages over iframes:
        - Sub-apps share global context; no URL or UI sync issues
    - Downsides:
        - No JS or CSS isolation
        - Requires lots of config changes (host + sub-apps); not out-of-the-box
5. **Qiankun**:
    - Open-sourced by Ant Group
    - Based on single-spa but more out-of-the-box
    - HTML-entry mode is as simple as iframes
    - Preloads unopened micro-frontend resources during browser idle time, speeding up navigation
    - Supports React, Vue, Angular, and other mainstream frameworks
    - Downside: style isolation between host and sub-apps isn't fully implemented — overrides/conflicts can still happen
        - Use a fixed format
        - CSS Modules
6. **Garfish**:
    - Open-sourced by ByteDance
    - Provides inter-app communication, style isolation, plugin system
    - Supports React, Vue, Angular, Ember, etc.
7. **MicroApp**:
    - Open-sourced by JD
    - Acts as glue between the **host app** and **sub-apps**
    - Focuses on communication and collaboration; rich APIs for events and data sharing
    - Supports mainstream frontend frameworks and provides visual dev/debug tools

### MicroApp vs Qiankun pros/cons

`micro-app` uses Web Components for sub-app isolation and communication.

**Pros**:

- Small footprint, fast load, minimal impact on host performance
- Simple config, easy to learn, good for rapid development
- Good compatibility — supports mainstream browsers

**Cons**:

- **Limited features**: fewer features; may need extra plugins or custom code for advanced needs



`qiankun` is a micro-frontend framework based on Single-SPA.

**Pros**:

- **Rich features**: style isolation, preloading, global state management
- **Strong style isolation**: CSS sandbox tech isolates sub-app styles reasonably well
- **Mature community and docs**: easy to learn and use

**Cons**:

- **Performance overhead**: rich features mean larger overhead; can impact host performance
- **Config complexity**: more complex setup; learning curve

### Qiankun global state management

Implemented via the `initGlobalState` method. The host app initializes global state, getting back a `MicroAppStateActions` instance with three methods:

- **onGlobalStateChange**: listens for global state changes. The callback receives the new state and previous state.
- **setGlobalState**: sets global state. Sub-apps can only modify existing top-level properties.
- **offGlobalStateChange**: removes the current app's listener. Sub-apps auto-call this on unmount.

### The `start` function in qiankun — purpose and parameters



- The main function to start the micro-frontend app
- Manages bootstrap, mount, unmount lifecycles for host and sub-apps
- Combines host and sub-apps into a complete micro-frontend application



1. **App registration**: `start` lets the host register sub-apps; they can be SPAs or traditional MPAs.
2. **Lifecycle management**: manages sub-app lifecycles — bootstrap, mount, unmount, etc.
3. **Routing**: `start` also handles routing between apps so sub-apps respond correctly to host route changes.
4. **Style isolation**: sandbox mechanism isolates sub-app styles to prevent conflicts.
5. **Global state management**: `start` can integrate global state so host and sub-apps share/manage state.

Options:

- **singular**: boolean — single-instance mode. If `true`, only one sub-app can be active at a time. Default: `false`.
- **fetch**: a function — customize how sub-app resources are loaded. Default: `window.fetch`.
- **sandbox**: an object — configure the sub-app sandbox; types include `strict`, `no-communication`, etc.
- **getTemplate**: a function — customize the sub-app mount-point template. Default: a simple `<div>`.
- **props**: an object — extra properties passed to sub-apps; the sub-app receives them via `props`.

### How qiankun handles JS pollution that the sandbox can't solve

The JS sandbox is implemented by proxying the `window` object — effectively isolating sub-app globals. But if onclick or addEventListener attaches a click event to `<body>`, that may still leak.

### micro-app

Uses CustomElement combined with custom Shadow DOM to wrap micro-frontends as Web Component-like components, achieving component-style rendering.

Thanks to Shadow DOM isolation, `micro-app` doesn't require sub-apps to modify render logic and expose methods (like single-spa or qiankun), nor modify webpack config — currently the lowest-cost micro-frontend solution.

```js
// my-page.js
export function MyPage () {
  return (
    <div>
      <h1>Sub-app</h1>
      // name (required): app name
      // url (required): app URL — auto-completed to http://localhost:3000/index.html
      // baseroute (optional): base route assigned by host to sub-app; here, `/my-page`
      <micro-app name='app1' url='http://localhost:3000/' baseroute='/my-page'></micro-app>
    </div>
  )
}

// Set base route (skip this step if host uses history routing and sub-app uses hash routing)
// router.js
import { BrowserRouter, Switch, Route } from 'react-router-dom'

export default function AppRoute () {
  return (
    // 👇 set base route; sub-app can get the host-provided baseroute via window.__MICRO_APP_BASE_ROUTE__. Empty string by default.
    <BrowserRouter basename={window.__MICRO_APP_BASE_ROUTE__ || '/'}>
      ...
    </BrowserRouter>
  )
}
```

### Lifecycle list

[1. created](https://cangdu.org/micro-app/docs.html#/zh-cn/life-cycles?id=_1-created)

Fires after `<micro-app>` is initialized, before resources load.

[2. beforemount](https://cangdu.org/micro-app/docs.html#/zh-cn/life-cycles?id=_2-beforemount)

Fires after resources load, before rendering starts.

[3. mounted](https://cangdu.org/micro-app/docs.html#/zh-cn/life-cycles?id=_3-mounted)

Fires after the sub-app renders.

[4. unmount](https://cangdu.org/micro-app/docs.html#/zh-cn/life-cycles?id=_4-unmount)

Fires when the sub-app unmounts.

[5. error](https://cangdu.org/micro-app/docs.html#/zh-cn/life-cycles?id=_5-error)

Fires on sub-app render errors — only fatal render-stopping errors trigger this.

### Data communication

- Sub-app getting data from host
    - `const data = window.microApp.getData()`
    - `window.microApp.addDataListener(dataListener: Function, autoTrigger?: boolean)`
    - `window.microApp.removeDataListener(dataListener: Function)`
    - `window.microApp.clearDataListener()`
- Sub-app sending data to host
    - `window.microApp.dispatch({type: 'data from sub-app'})`
- Host sending data to sub-app
    - micro-app's `data` attribute
- Manual data send
    - `microApp.setData`
- Host getting data from sub-app
    - `microApp.getData(appName)`
- Global data communication
    - Send global data `microApp.setGlobalData({type: 'global data'})`
    - `microApp.addGlobalDataListener(dataListener: Function, autoTrigger?: boolean)` // unbind listener `microApp.removeGlobalDataListener(dataListener: Function)` // clear all global listeners bound by host `microApp.clearGlobalDataListener()`

- **URL parameters**: pass data via browser URL query parameters.



### Qiankun's mechanisms for JS and CSS isolation

1. JavaScript isolation

Qiankun provides two sandbox modes:
- **Snapshot sandbox**: for browsers without Proxy. Saves/restores snapshots of the global object (`window`) for isolation.
- **Proxy sandbox (LegacySandbox and ProxySandbox)**: uses ES6 Proxy to intercept access/modification of `window`, giving each sub-app its own environment.

**How Proxy sandbox works**:
- Creates a fake `window` object (via Proxy).
- When the sub-app accesses or modifies properties on `window`, the operation is proxied into the sub-app's own sandbox.
- On unmount, the sandbox cleans up sub-app globals and event listeners to avoid polluting the host or other sub-apps.

2. CSS isolation

**Scoped CSS**

- Sub-app stylesheets are dynamically added to the DOM at load and removed at unmount.
- Use `scoped` attribute or CSS Modules to ensure styles only apply to the sub-app's DOM.

**Shadow DOM**

Render the sub-app into a **Shadow DOM** to leverage its natural isolation, ensuring no leakage to other apps.

**Dynamic stylesheet add/remove**

On sub-app load, sub-app stylesheets are inserted; on unmount, they are removed.

3. Other isolation mechanisms

Beyond JS and CSS isolation, Qiankun also ensures sub-app independence via:
- **Global event isolation**: by overriding `addEventListener` and `removeEventListener` so sub-app listeners don't affect other apps.
- **Timer isolation**: by hijacking `setTimeout` and `setInterval` so timers are cleared on sub-app unmount.

### Qiankun's communication modes

Communication between host and sub-app (cross-app communication)

1. **Pass data via props (nested communication)**: Qiankun lets you pass data via props to sub-apps. For example, the host can pass config or data to sub-apps, which they receive on mount.

   ```javascript
   // Host passes data to sub-app
   const app = {
     name: 'app-vue',
     entry: 'http://localhost:7100',
     props: {
       customData: 'Hello, Sub-App',
     }
   };
   ```

2. **Via qiankun's global event mechanism (event communication)**: Qiankun provides `setGlobalState` and `onGlobalStateChange` for sharing global state or events across sub-apps:

    - `setGlobalState(state)`: update global state.
    - `onGlobalStateChange(callback)`: listen to global state changes.

   ```javascript
   // In the host: set global state
   window.__POWERED_BY_QIANKUN__ && window.setGlobalState({
     user: 'John',
     role: 'admin'
   });

   // In sub-app: listen to global state
   window.__POWERED_BY_QIANKUN__ && window.onGlobalStateChange((state, prevState) => {
     console.log('Global state changed:', state);
   });
   ```

1.2 Communication between sub-apps

1. **`setGlobalState` and `onGlobalStateChange`**: sub-apps can also communicate via these APIs. Sub-app A updates global state; sub-app B receives the change notification. Useful when sub-apps need to share data or stay in sync.
2. **Custom events**: sub-apps can also use `CustomEvent` for messaging. Sub-app A dispatches an event; sub-app B listens for it.

### **Qiankun vs Iframe**

- **Qiankun** — for **micro-frontend architectures**; uses shared state, events, props for host–sub-app communication. Sub-apps can be any frontend framework (React, Vue, etc.) and usually run in the same JS environment.
- **Iframe** — a browser mechanism for embedding pages. Each iframe is an independent browser context; host–iframe communication is typically via `postMessage`.
