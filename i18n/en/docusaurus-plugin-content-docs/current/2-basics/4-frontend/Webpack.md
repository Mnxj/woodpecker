## Basic usage

Webpack is a `static asset bundler`.

Outputs compiled files (bundles) that browsers can run directly.

## Features

Webpack itself is limited:

- Dev mode: only compiles `ES Module` syntax in JS
- Prod mode: compiles ES Modules and minifies JS

## Getting started

### 1. Source structure

```js
// ./src/main.js
import count from './js/count'
import sum from './js/sum'

console.log(count(2,1))
console.log(sum(1,2,3,4))

// ./src/js/sum.js
export default function count(x,y) {
    return x-y;
}
// ./src/js/count.js
export default function sum(...args) {
    return args.reduce((p,c) => p+c, 0)
}
// ./public/index.html
<h1> hello webpack</h1>
<script src="../src/main.js"></script>
```

> Opening index.html shows nothing logged.

```shell
# init package.json
npm init -y

# install
npm i webpack webpack-cli -D

# build
npx webpack ./src/main.js --mode=development

# ./public/index.html
  <script src="../dist/main.js"></script>
```

> The log will show `eval` hints — dev only compiles JS syntax, no minification.

```shell
# Production mode has no eval hints
```

### chunkHash vs contentHash in Webpack

> Control file caching effectively to improve loading speed and performance.

**Meaning**

- `chunkHash`: hash based on the entire chunk's content — for `dynamic imports` or chunks generated by code splitting. With `import()` dynamic import, the module is placed in its own chunk.
- `contentHash`: hash based on the file content — for static assets that don't change during build. Keeps the hash stable as long as the content is unchanged.

**Example config**

```js
module.exports = {
  // ...
  output: {
    // Use [chunkhash] or [contenthash] placeholders to insert the hash
    filename: '[name].[chunkhash].js',
    chunkFilename: '[name].[chunkhash].chunk.js',
  },
  // ...
};
```

### Webpack loaders and plugins

**Loader**:

- Essentially a function
- Handles transformations for non-JS files (CSS, images, fonts, etc.)
- Chainable — bottom to top, right to left
- Defined in `module.rules`
- Common loaders:
    - `babel-loader`: transpile ES6+ to ES5
    - `css-loader`: process CSS files
    - `file-loader`: process images, fonts, etc.
    - `sass-loader`: transpile Sass to CSS

**Plugin**:

- JS objects with an `apply` method
- Access Webpack's full compile lifecycle
- Can modify any Webpack option
- Solve custom needs loaders can't — file compression, asset optimization
- Defined in the `plugins` array
- Common plugins:
    - `HtmlWebpackPlugin`: auto-generate HTML files
    - `MiniCssExtractPlugin`: extract CSS from the bundle into separate files
    - `OptimizeCssAssetsWebpackPlugin`: optimize and minimize CSS assets
    - `webpack.HotModuleReplacementPlugin`: enable HMR

### How does Webpack resolve `import`?

1. On `import`, use the module resolver.
2. Resolve the module identifier in the `import` statement to a relative or absolute path.
3. Find the module file based on the path.
4. Load and execute the module file's code.
5. While loading, also resolve internal `import` statements — recursively handle the module dependency graph and build the dep graph.
6. Finally, based on the dep graph, bundle all modules into one or more output files.

### Vite vs Webpack

1. **Dev server**
    - **Vite** uses native browser ESM support for instant HMR.
    - **Webpack** uses webpack-dev-server — requires bundling before serving.
2. **Build process**
    - **Vite** uses esbuild for bundling — much faster than Webpack. Written in Go.
    - **Webpack** is written in JS — longer build times, especially on large projects.
3. **Dependency pre-bundling**
    - **Vite** uses `esbuild` for pre-bundling — dependencies bundled into single files for faster dev-server startup.
    - **Webpack** re-parses and re-bundles all deps on every build.
4. **Configuration complexity**
    - **Vite** — simple config; focused on dev and build.
    - **Webpack** — more complex; handles code splitting, lazy loading, more features.
5. **Plugin ecosystem**
    - **Webpack** has a richer ecosystem than **Vite**.

### Rollup vs Webpack

- **Design philosophy**: Rollup focuses on ES6 module bundling; Webpack is a comprehensive bundler supporting many assets and complex build workflows.
- **Output formats**: Rollup supports many output formats but focuses on ES modules; Webpack supports ES modules, CommonJS, UMD, etc.
- **Use cases**: Rollup is great for building libraries and frameworks; Webpack is great for complex applications.

### Module loading

1. Determine the entry from config.
2. Analyze the entry module and its deps; build a dep graph.
3. Async-load required modules based on the graph.
4. Bundle all modules into one or more files containing the deps.
5. At runtime, dynamically load and execute these files based on load order; cached modules are reused.

### Do you know AST?

- Abstract Syntax Tree — an intermediate representation a compiler or interpreter builds when compiling or interpreting source code.
- Each node is a syntactic construct; relationships between nodes represent syntactic structure and logic.
- AST analysis enables linting, optimization, and code generation.

### How is an AST built?

Lexical analysis: a lexer breaks source code into tokens (the smallest meaningful units).

Syntactic analysis: the parser checks syntax rules and builds a tree — the AST.

### What is Babel? How does it work?

A JavaScript compiler that lets developers use the latest JS (ES6+).

```js

// AST => transformed AST

// 3. @babel/generator turns the modified AST back into a code string and produces a source map.
```

### ESLint

A static analysis tool for spotting patterns and errors in JS code. Highly configurable to fit project-specific style and quality standards.



Basic setup:

a. Install ESLint:

```shell
npm install eslint --save-dev
```

a. Initialize the config: run `eslint --init` to generate a config file.



b. Configure rules: in `.eslintrc`, define your rules — severity, environments (browser, Node, etc.), globals, plugins, extends. Rules can be `"off"`, `"warn"`, or `"error"`.

c. Integrate into the build: add ESLint to your build scripts via npm scripts or your task runner.



### How ESLint works

1. Config — set up rules.

2. Parsing — when ESLint runs, a parser (e.g. `espree`, the default) parses the code into an AST. ESLint traverses the AST, checking each node against applicable rules. Violations are recorded.

3. Reporting — after traversal, ESLint generates a report listing all issues, classified as errors or warnings.

4. Fix — ESLint can auto-fix when possible. `editor.codeActionOnSave: { source.fixAll: true }`.

5. Integration — integrates with IDEs for live feedback during coding, and with build tools like Webpack.

### How to enable persistent caching

• Set HTTP cache headers server-side (cache-control)

• Bundle deps and runtime into separate chunks (splitChunk) — they rarely change

• Lazy-load via `import()` — dynamically-loaded files become independent chunks with their own chunkhash

• Keep hashes stable — compilation and file content changes shouldn't disturb other files' hashes (especially for low-version targets)

### Webpack lifecycle

Initialization phase

1. Read config from the config file (default `webpack.config.js`) or CLI args.
2. Create a Compiler instance based on config — responsible for the entire bundling process.
3. Load plugins specified in config; run their `apply` methods to attach to the Compiler.

Compilation phase

1. The Compiler starts compiling — creates a Compilation object representing a single compile run.
2. Starting from the entry file, recursively build the module dep graph. Each module is transformed by configured loaders.
3. After modules are built, Webpack generates chunks.

Output phase

1. **Output assets**: based on configured output path and filenames, write assets to the target dir.
2. **Finish**: all assets written; bundling done.

### Lifecycle hooks

- `entry-option`: after reading config, before initializing Compiler.
- `after-plugins`: after all built-in and user plugins are added to Compiler.
- `compiler.hooks.run`: when Compiler starts.
- `compiler.hooks.compile`: when a new compilation object is created.
- `compilation.hooks.optimize`: at the start of optimization.
- `compilation.hooks.afterOptimizeAssets`: after optimization completes.
- `compilation.hooks.processAssets`: during asset processing.
- `emit`: before assets are emitted to the output dir.
- `done`: when all compilation is complete.

### tapable

A small library providing Webpack's hooks system.

Main uses:

1. **Plugin system**: build your own plugins.
2. **Modularization**: via `tapable`, Webpack's compilation is modularized into stages — each can be extended via hooks. Cleaner and more maintainable.
3. **Performance**: lets devs precisely control what runs and what's skipped.

### tapable design and implementation

**Design**

`tapable` is designed around:

1. A set of hook classes letting devs insert custom logic at different stages of Webpack's compilation — modular and pluggable.

2. Sync hooks, async hooks, parallel hooks, etc.

3. Via hooks, only necessary logic runs during compilation.



**Implementation**

`tapable`'s implementation:

1. Each hook class encapsulates logic for registering and triggering callbacks.

2. Call the hook's `tap` method to register a callback. `tap` takes a name (optional) and the callback.

3. When you want to fire registered callbacks, call `call`, `callAsync`, or `promise`.

4. Execution depends on the hook type. Sync hooks run callbacks sync; async hooks run async — parallel or serial.

5. Supports hook chaining — one hook can trigger another, forming a chain.

```js
const { SyncHook } = require("tapable");
class MyWebpackPlugin {
  constructor(options) {
    // Plugin options
    this.options = options;
  }
  apply(compiler) {
    // Create a sync hook
    const hook = new SyncHook(["compilation"]);
    // Register a callback
    hook.tap("MyPlugin", (compilation) => {
      console.log("MyPlugin: Compilation started!");
    });

    // Register hook into Webpack lifecycle
    compiler.hooks.compile.tap("MyPlugin", (compilation) => {
      hook.call(compilation);
    });
  }
}

module.exports = MyWebpackPlugin;
```

### CDN principles

**Distributed storage**: many server nodes across geographies.

**Caching**: when a user requests a resource, the nearest node checks its cache. If cached, serve directly — reduces transfer distance and time, faster access.

**Smart routing**: routes user requests to the nearest, best-performing node based on geography and network conditions.

**Content updates**: when content changes, the origin pushes updates to the CDN; nodes update their caches.

**Load balancing**: distributes user requests across nodes — avoids single-server overload.

### How Webpack HMR works

1. **Compiler**: the Webpack compiler generates an HMR update manifest with module IDs and updated hashes.
2. **Runtime**: HMR runtime is injected into the entry. It communicates with the compiler, receives updates, and decides which modules to update based on the manifest.
3. **Module updates**: when an update is detected, the runtime requests updated code from the compiler. The compiler sends the new module code.
4. **Module replacement**: after receiving the new code, a strategy replaces the old module. CSS modules can replace stylesheets directly. JS modules use hot-replacement logic.
5. **Notify the app**: notify which modules updated; custom logic can run.

### Async loading principles

- Dynamic `import()`: `import().then`

- `require.ensure(, () => ...)` — async load modules into a specific chunk.

- `bundle-loader`: place modules in a separate file, loaded on demand.



**How Webpack implements dynamic loading**

When Webpack encounters a dynamic import, it doesn't bundle it into the main file — it bundles it as a separate file. At runtime, when the module is needed, an HTTP request loads the file.

Reduces main file size and improves load speed.

Webpack can also code-split and lazy-load dynamic modules — further optimizing perf. In browsers that support dynamic imports, use `import()`. In unsupported browsers, use Webpack's `require.ensure` or `require.include` for async loading.



### What perf issues does `require` have?

1. **Synchronous blocking**: in traditional CommonJS, `require` is synchronous — a slow module load blocks subsequent code.
2. **Overhead with many modules**: heavy `require` use across many modules can incur overhead in complex apps.

### `require` caches loaded modules in memory — what problems can this cause?

**Memory usage**: many unused cached modules can gradually eat memory, especially in long-running apps.

**Version conflicts**: when the same module is loaded in different places expecting different versions, caching may always use the first-loaded version — causing inconsistency.

### Proxy principles

Workflow:

1. Browser sends a request to the proxy server.
2. Proxy receives it; decides how to handle based on config.
3. Forwards the request to the target server.
4. Target server processes and returns a response to the proxy.
5. Proxy may modify the response before returning it to the client.

Types:

- **Forward proxy**: on the client side; configure the proxy address to access external networks. Hides client real IP for anonymous access.
- **Reverse proxy**: on the server side; clients directly access the reverse proxy, which routes requests to backend servers. Reverse proxies enable load balancing, caching, security filtering.

### How does `sideEffects` help Webpack optimize?

Webpack checks the module's `sideEffects` field. If a module has `sideEffects: false`, Webpack considers it side-effect-free. During tree shaking, unused exports are removed — they won't affect the project's behavior.

### AMD (Async Module Definition) vs CMD (Common Module Definition)?

Used to define modules and dependencies for modular development.

Differences

- **Dep declaration**: AMD declares all deps at module definition time; CMD loads deps lazily as needed.
- **Loading timing**: AMD pre-loads deps; CMD lazy-loads.
- **Exports**: AMD modules return values for exports; CMD modules use `exports` object or `module.exports`.

### HappyPack shared process pool setup

Speeds up JS loading and compilation by breaking tasks into parallel child processes.

1. Install

   ```bash
   npm install --save-dev happypack
   # or
   yarn add happypack --dev
   ```

2. **Import HappyPack**:

   ```javascript
   const HappyPack = require('happypack');
   ```

3. **Create a shared process pool**:
   Use HappyPack's `ThreadPool` to create a shared pool. Pool size usually equals CPU core count:

   ```javascript
   const os = require('os');
   const happyThreadPool = HappyPack.ThreadPool({ size: os.cpus().length });
   ```

4. **Configure HappyPack plugin**:
   Add HappyPack plugin instances to webpack's `plugins` array. Specify a unique id per file type and the loaders:

   ```javascript
   plugins: [
     new HappyPack({
       id: 'js', // identifies which file type this HappyPack instance handles
       loaders: ['babel-loader?cacheDirectory'], // specify loaders
       threadPool: happyThreadPool, // use shared pool
       verbose: true, // allow HappyPack output
     }),
     // Add more HappyPack instances for different file types
   ]
   ```

5. **Modify rules to use HappyPack**:
   In webpack's `module.rules`, replace loaders with `happypack/loader` and specify the id:

   ```javascript
   module: {
     rules: [
       {
         test: /\.js$/, // match .js files
         use: ['happypack/loader?id=js'], // use HappyPack for .js
         include: path.resolve('src'), // include this dir
         exclude: /node_modules/, // exclude node_modules
       },
     ]

   ```

HappyPack has been replaced by `thread-loader` in Webpack 5.

### How to use thread-loader?

`thread-loader` enables multi-thread processing similar to HappyPack. It places specific loaders in a separate Node.js process.

Basic steps:

1. **Install thread-loader**:

   ```bash
   npm install --save-dev thread-loader
   # or
   yarn add thread-loader --dev
   ```

2. **Configure thread-loader**:

   ```javascript
   const path = require('path');

   module.exports = {
     // ...
     module: {
       rules: [
         {
           test: /\.js$/,
           include: path.resolve('src'),
           exclude: /node_modules/,
           use: [
             {
               loader: 'thread-loader',
               options: {
                 // worker options
                 // e.g. worker count
                 workers: require('os').cpus().length,
               },
             },
             'babel-loader',
           ],
         },
         // other rules...
       ],
     },
     // ...
   };
   ```

   Use for CPU-heavy loaders like `babel-loader`, `ts-loader`. For lightweight loaders, the overhead may negate gains.

### Webpack build-speed optimization (with umi/max examples)

1. webpack4 — use chunk caching.

- webpack-bundle-analyzer: `pnpm run analyze`

- `optimization.splitChunks.cacheGroups.vendors` tells webpack: any module in `/node_modules` referenced ≥ 2 times goes into a single `vendors` file instead of being split across page files.

This extracts common deps into a single file shared by pages.

After that, remember to add the extracted `vendors` file to `chunks`. `chunks` lists files needed at initial load. `vendors` contains all third-party libs and is always needed at load time.

Saves about 1MB:

```js
chainWebpack: function (config, { webpack }) {
    config.merge({
      minimize: true,
      optimization: {  // webpack config
        splitChunks: {
          chunks: 'all',
          minSize: 20000,
          minChunks: 2,
          cacheGroups: {
            vendors: {
              name(module) {
                const reg = /(echarts|zrender|brace|react-json-view)/;
                if (reg.test(module.identifier())) {
                  const [chunkName] = reg.exec(module.identifier()) || [];
                  return `npm.${chunkName}`;
                };
                return 'vendors';
              },
              test({ resource }) {
                return /[\\/]node_modules[\\/]/.test(resource);
              },
              priority: 10,
            },
          },
        },
      },
    });
```

2. webpack 5

Default-enabled webpack 5 [cache](https://link.juejin.cn/?target=https%3A%2F%2Fwebpack.js.org%2Fconfiguration%2Fcache%2F%23root) feature also helps build speed.

`umi.js`'s third-party deps (`/node_modules`) can't be split by `optimization.splitChunks`.

`umi.js` contains many rarely-updated deps. When we change app code and rebuild, `umi.js`'s hash often changes — so users need to re-download `umi.js` (and its unchanged deps) every release.

Stable, project-essential deps like `antd` / `moment` / `react` / `react-dom`.

3. externals

- `externals` excludes some third-party libraries from the bundling process — they don't appear in the final output. They're loaded another way (e.g. `<script>` tag).

- `externals` is an object: keys are library names; values are the global variable names from the `<script>` tag.
- Putting `antd` / `moment` / `react` / `react-dom` / `xlsx` in `externals` and loading from CDN excludes them from `umi.js`. Users with cached files don't re-download them on new releases.

```js
import * as antd from 'antd';
import _ from 'lodash';

let externalsAntd = {};
for (const key in antd) {
  if (Object.hasOwnProperty.call(antd, key)) {
    externalsAntd[`antd/es/${_.kebabCase(key)}`] = `antd.${key}`;
  }
};

const config = {
  // ...other umi config
  externals: {
    react: 'React',
    moment: 'moment',
    xlsx: 'XLSX',
    'react-dom': 'ReactDOM',
    'moment/locale/zh-cn': 'moment.locale',
    ...externalsAntd,
  },
  styles: ['https://cdnjs.cloudflare.com/ajax/libs/antd/4.23.6/antd.min.css'],
  scripts: [
    'https://cdnjs.cloudflare.com/ajax/libs/react/16.14.0/umd/react.production.min.js',
    'https://cdnjs.cloudflare.com/ajax/libs/react-dom/16.14.0/umd/react-dom.production.min.js',
    'https://cdnjs.cloudflare.com/ajax/libs/moment.js/2.29.4/moment.min.js',
    'https://cdnjs.cloudflare.com/ajax/libs/antd/4.23.6/antd.min.js',
    'https://cdnjs.cloudflare.com/ajax/libs/moment.js/2.30.1/locale/zh-cn.min.js',
    'https://cdnjs.cloudflare.com/ajax/libs/xlsx/0.17.5/xlsx.full.min.js',
  ],
};
```

`umi.js` shrinks noticeably, but first-time user load might not decrease — possibly even increase. **Why? Because `externals` deps loaded via CDN don't pass through webpack, so they don't support webpack's tree-shaking.** For deps that can be split via `optimization.splitChunks` and support tree-shaking, don't put them in `externals`.

Even if first load doesn't shrink, CDN-loaded deps don't need to be re-downloaded on subsequent releases. That's the value of `externals`.

4. runtimeChunk

If you change a page's code and rebuild, both `umi.js` and `Tracking_detail.js` hashes change. You modified one page but users must re-download `umi.js` + `Tracking_detail.js`.

This is because by default, `umi.js` contains the whole app's [runtime and manifest](https://link.juejin.cn?target=https%3A%2F%2Fwebpack.js.org%2Fconcepts%2Fmanifest%2F%23runtime). `runtime and manifest` includes hashed filenames. Any file's hash change updates `umi.js`'s content and hash — invalidating cached `umi.js`.

If we extract `runtime and manifest` into a separate file, changing single page code requires only re-downloading that page's JS file + a tiny `runtime.js`, while using cached `umi.js`.

Extract via webpack's `optimization.runtimeChunk`:

```js
{
  // ...other umi config
  chunks: ['runtime~umi', 'vendors', 'umi'], // add 'runtime~umi'
  chainWebpack: function (config, { webpack }) {
    config.merge({
      optimization: {
        runtimeChunk: true, // add runtimeChunk
        splitChunks: {
          // ...splitChunks config
        },
      },
    });
  },
};
```
