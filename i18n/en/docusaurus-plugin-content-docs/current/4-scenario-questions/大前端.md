### What is Single Sign-On (SSO) and what is the flow?

1. **User accesses application A**
2. Application A finds the user is not logged in and redirects to the SSO server.
3. On the SSO server, the user logs in with credentials (username, password, etc.).
4. The SSO server verifies the credentials.
5. After verification, the SSO server creates a global token.
6. **Returns the token to application A**
7. **Application A validates the token**
8. **User accesses application A's resources**
9. When the user later visits application B (which also trusts this SSO server), application B fetches and validates the token from the SSO server.
10. Once validated, the user can access resources in application B without logging in again.

Benefits of SSR include:

1. **Faster first-screen load**: Since the server already produces a complete HTML page, the browser can display content faster, reducing client-side rendering time.
2. **Better SEO**: Search engine crawlers can directly read the server-rendered HTML, making the page easier to understand and index, improving search ranking.
3. **Improved user experience**: Users see content sooner, with less waiting.

However, SSR also has challenges:

1. **Server performance demands**: The server has to handle more requests and rendering work, requiring stronger server resources.
2. **Increased development complexity**: You need to consider both server-side and client-side development.
3. **Data fetching and synchronization**: Data fetched on the server must stay consistent with the client to avoid inconsistencies.
4. **Caching strategy**: Sensible caching is needed to improve performance and reduce server load.

### Difference between MVVM and MVC

1. **Component responsibilities**:
   - MVC: Model handles data and business logic, View handles UI display, Controller processes user input and updates Model/View.
   - MVVM: Model handles data and business logic, View handles UI, ViewModel handles communication between Model and View.
2. **View-Controller relationship**:
   - MVC: View and Controller are tightly coupled; the Controller directly manipulates the View.
   - MVVM: ViewModel is loosely coupled with the View through data binding; the View only renders.
3. **Data flow**:
   - MVC: Data flow is one-way.
   - MVVM: Data flow is two-way.
4. **Testability**:
   - MVC: The Controller is typically hard to test because it depends on both Model and View.
   - MVVM: The ViewModel can be tested independently of the View since it doesn't depend on a specific View implementation.

### Evolution of frontend development

1. **Static pages (1990s):**
   - Mainly static HTML; content and styles were in the same file.
   - Interactivity was minimal; most features relied on backend services.
2. **Dynamic pages (2000s):**
   - With JavaScript, developers could build simple interactions in the browser.
   - Server-Side Rendering (SSR) emerged — the backend generated HTML and returned it to the client.
3. **Single Page Apps (SPA) era (2010s):**
   - For richer UX, SPAs based on JavaScript frameworks/libraries emerged.
   - The frontend manages page state and interaction logic; the backend provides APIs.
   - Representative frameworks: React, Vue.js, Angular.
4. **Full-stack development (2010s to present):**
   - Frontend developers increasingly need backend skills, becoming full-stack.
   - JavaScript-based runtimes like Node.js let frontend and backend share code.
   - Frontend frameworks started integrating backend services, forming "full-stack frameworks" like Next.js, Nuxt.js.
5. **Micro-frontends (2010s to present):**
   - To tackle the complexity of large SPAs, micro-frontend architecture appeared.
   - Apps are split into multiple independent micro-frontend modules, each developed, deployed, and maintained independently.
   - Representative solutions: single-spa, Module Federation.
6. **Low-code / no-code (2010s to present):**
   - To lower the barrier of frontend development, various low-code / no-code platforms emerged.
   - Developers can build apps quickly by dragging components and configuring parameters.
   - Representative platforms: Bubble, Appian, Outsystems.

### What can you do to reduce requests when opening a page?

1. **Asset bundling and compression**:
   - Merge CSS and JavaScript files to reduce HTTP requests.
   - Compress CSS, JS, and images to reduce file sizes.
2. **Use a CDN**:
   - Deploy static assets (CSS, JS, images) to CDN servers for proximity-based delivery.
3. **Defer and lazy loading**:
   - Load resources only when needed to reduce initial load.
   - For example, load images when the user scrolls to them.
4. **Caching strategy optimization**:
   - For static assets, set proper HTTP cache headers to leverage browser cache.
   - For dynamic resources, use Etag or Last-Modified for incremental updates.
5. **SSR or SSG**:
   - Use Server-Side Rendering or Static Site Generation to pre-render pages on the server, reducing client requests.
6. **HTTP/2 or HTTP/3 optimization**:
   - Use HTTP/2 or HTTP/3 for multiplexing, header compression, and reduced TCP overhead.
7. **Image optimization**:
   - Pick appropriate image formats (WebP, AVIF) and compression to reduce file sizes.
   - Use `srcset` and `sizes` to serve different image sizes for different screens.
8. **Skeleton screens / preloading**:
   - Use skeleton screens to show placeholders during loading.
   - Use `<link rel="preload">` to preload critical resources, reducing blocking in the critical rendering path.
9. **API optimization**:
   - Batch requests, incremental updates to reduce API call counts.
   - Cache infrequently-changing data appropriately.

### What performance bottlenecks exist in frontend projects, and how do you diagnose and resolve them?

• Diagnostic tools: Use Chrome DevTools, Lighthouse, or WebPageTest.

• Code splitting and lazy loading: Use dynamic `import()` to reduce initial load time.

• Image and asset optimization: Compress images, use modern formats (WebP), and apply caching.

• Reduce repaints and reflows: Optimize CSS properties (avoid those that trigger reflow); use `transform` and `opacity` for animation since they don't trigger reflow.

• Use Web Workers: For complex data processing, move work off the main thread to avoid UI freezes.

### Describe a major frontend incident — how did you locate and resolve it?

Example frontend incident handling process:

• Problem: Users report that a specific page loads abnormally slowly.

• Locate the problem:

a. Performance profiling: Record and analyze page load with Chrome DevTools' Performance tab.

b. Check resource loading: Inspect the Network tab to see if any resources are slow or failing.

c. JS profiling: Use the JS Profiler to find functions taking too long.

d. Code review: Review the relevant code for inefficient operations, such as complex DOM work inside loops.

• Solve the problem:

a. Image optimization: Found some large images unoptimized — used compression tools and converted to more efficient formats (WebP).

b. Code optimization: Found unnecessary repeated DOM queries and updates in JS — cached DOM refs and reduced DOM operations.

c. Lazy loading: Lazy-loaded non-critical resources to speed up first paint.

• Follow-up: After fixing, used code review and performance monitoring to continuously track performance and prevent future issues.

### What processes does a browser have?

- Browser main process: Manages the browser UI — address bar, bookmarks, back/forward buttons. Also handles invisible background tasks like network requests, file access, and cross-process communication.
- Renderer process: Handles content of browser tabs.
- Plugin process: For browsers using plugins (like Flash Player), each plugin instance typically runs in its own process.
- GPU process: Handles graphical tasks — 2D and 3D drawing. Translates render-process drawing commands into formats the GPU understands and sends them to the GPU.
- Network process: Handles network requests (HTTP), connections, caching, and security checks.
- Storage process: Handles file system access and database operations like IndexedDB, LocalStorage, and Cookies.
- Extension process: For browsers with extensions installed, each extension typically runs in its own process — isolating extension code so a crash doesn't affect the rest of the browser.
- Service Worker process: For offline caching, background sync, etc.

### How does a browser work internally?

1. **Rendering engine**
   - Parses HTML, CSS and renders a visual page.
   - Includes: parser, DOM tree construction, style calculation, layout, paint.
2. **JavaScript engine**
   - Interprets and executes JavaScript.
   - Includes: syntax analysis, interpretation, memory management.
3. **Event loop**
   - Coordinates JS single-threaded execution.
   - Continuously checks the task queue and processes tasks in FIFO order.
   - Covers sync tasks, microtasks, macrotasks.
4. **CORS (Cross-Origin Resource Sharing)**
   - The browser restricts cross-origin access for security.
   - Uses response headers to allow or deny cross-origin requests.
   - The server must cooperate.
5. **Caching**
   - The browser caches static assets — images, CSS, JS, etc.
   - Cache policy controlled via HTTP headers.
   - Speeds up page loads and reduces server load.
6. **Browser extensions (plugins)**
   - Browsers provide extension APIs.
   - Extensions add features like ad blocking, password managers.
   - The extension mechanism differs slightly per browser.
7. **Sandboxing**
   - Browsers run web pages in sandboxed environments for security.
   - Pages can't access the local file system or sensitive resources.
   - Sandboxing prevents XSS, CSRF, etc.

### How do you view mini-programs, and what's the technical principle?

A mini-program is an app that can be used without download/install.

A few aspects:

1. Based on web technologies, but can also call native phone features.
2. Developers can use existing web dev skills and tools.
3. Mini-program code is compiled and packaged before release into a format the platform recognizes.
4. Mini-programs run in an isolated environment that provides required APIs and components while ensuring safety and performance.
5. Updates are delivered via the cloud — when developers release a new version, users automatically load the latest on next open, no manual update needed.

### Why do mini-programs have two threads?

Take WeChat mini-programs as an example — they use a "dual-thread model":

- **Render layer (View)**: Renders the UI, runs in WeChat's built-in browser, uses WXML and WXSS (web-like).
- **Logic layer (App Service)**: Handles business logic and data, runs in WeChat's JS engine, can call mini-program APIs for network requests, storage, etc.

### What does the server need for JSONP?


### How do you understand data-driven views and what are the core elements?

Emphasizes that views (the UI display) should be driven directly by the data model.

1. The data model is the source of truth.
2. Any change to the data model should automatically trigger a view update.
3. In data-driven views, view rendering is usually declarative. Developers declare how the view should render based on the data model rather than imperatively controlling the render process.
4. **Componentization**: Break views into reusable components, each responsible for rendering part of the data model. Improves maintainability and reusability.
5. **State management**: For complex apps, you need a mechanism to manage state. State management libraries (Redux, Vuex) help share and manage state across components.
6. In data-driven views, data flow is typically one-way.

### How much do you know about DNS?

A distributed network directory service that translates human-friendly domain names into IP addresses that computers understand.

How it works:

1. **Query process**: When a user tries to access a domain, the device sends a query to the configured DNS server.
2. **Recursive query**: If the DNS server doesn't have a cached record, it queries the root DNS server, which points to the TLD server, which points to the authoritative server, until the IP is found.
3. **Caching**: Each DNS server caches results for a time to speed up subsequent queries.
4. **Response**: The authoritative DNS server returns the IP to the user device, which uses it for communication.

### How to do DNS prefetching?

Pre-resolve domains the page might need to reduce DNS resolution latency when the user follows a link.

1. **HTML meta tag**:
   In the `<head>` of the HTML, add a `<meta>` tag to enable DNS prefetching.

   ```html
   <meta http-equiv="x-dns-prefetch-control" content="on">
   ```

   This tells the browser to enable DNS prefetching. `content="on"` enables it, `content="off"` disables.

2. **Link tag**:
   Use a `<link>` tag with `rel="dns-prefetch"` to specify the domain to prefetch.

   ```html
   <link rel="dns-prefetch" href="http://example.com">
   ```

   This tells the browser to DNS-prefetch `http://example.com`. You can include multiple `<link>` tags for different external resources.

3. **HTTP header**:
   On the server, enable DNS prefetching via an HTTP response header.

   ```
   X-DNS-Prefetch-Control: on
   ```

   This tells the browser to enable DNS prefetching. Same `on`/`off` semantics.

### If JS needs to run 1,000,000 tasks, how to keep the browser smooth?

To keep the browser smooth while running many tasks:

- **Chunking**: Split tasks into smaller batches and run them using `requestIdleCallback` or `setTimeout` to avoid blocking the main thread.
- **Web Workers**: Offload CPU-intensive tasks to a Web Worker so the main thread isn't blocked.
- **Async programming**: Use `Promise`, `async/await` to avoid blocking the main thread.

```javascript
function processTasks(tasks) {
    let index = 0;
    function run() {
        if (index >= tasks.length) return;
        requestAnimationFrame(() => {
            tasks[index]();
            index++;
            run();
        });
    }
    run();
}
```

### How to render 100,000 records at once without lag?

Use virtual lists — only render data in the visible region.

```javascript
function renderList(data) {
    const container = document.getElementById('list');
    const visibleCount = Math.ceil(container.clientHeight / 30);
    const [start, setStart] = useState(0);

    const visibleData = data.slice(start, start + visibleCount);

    return (
        <div onScroll={(e) => setStart(Math.floor(e.target.scrollTop / 30))}>
            {visibleData.map(item => <div key={item.id}>{item.name}</div>)}
        </div>
    );
}
```

### Are there risks to constantly adding things to `window`?

1. **Name collisions**: Adding variables or functions to the global scope can collide with existing globals or third-party libraries, causing unexpected behavior or errors.
2. **Memory leaks**: Objects attached to `window` — especially large objects or those with event listeners — can leak memory if not cleaned up.
3. **Hard to maintain**: Globals make code harder to maintain and understand, especially in large projects. Changing a global can affect other parts of the code, causing hard-to-track bugs.
4. **Security issues**: Exposing sensitive data or functionality in the global scope can be a security risk. Malicious scripts can access or modify these via globals.
5. **Performance impact**: Too many globals enlarge the global object and may hurt performance.

### What is OAuth 2.0 as a login flow?

Allows third-party apps (clients) to gain limited access to server resources via a user agent (typically a web browser).

1. **User-agent redirect**: The user clicks "log in" in a third-party app, which redirects the user agent (browser) to the authorization server.
2. **User authentication**: The user authenticates on the authorization server, typically by entering username and password.
3. **Authorization**: After authentication, the authorization server asks whether the user agrees to let the third-party app access their data. On consent, the authorization server issues an authorization code.
4. **Get access token**: The third-party app uses the authorization code to request an access token from the authorization server.
5. **Access resources**: The third-party app uses the access token to request the user's data from the resource server.

### Inner workings of XHR — write AJAX by hand

```js
function ajax(url, method, callback) {
  // Create XHR
  var xhr = new XMLHttpRequest();

  // Configure request
  xhr.open(method, url, true);

  // Set the onreadystatechange callback
  xhr.onreadystatechange = function() {
    // When request completes
    if (xhr.readyState === XMLHttpRequest.DONE) {
      // Check HTTP status
      if (xhr.status === 200) {
        // Success — invoke callback
        callback(xhr.responseText);
      } else {
        // Failure — invoke callback with error
        callback('Error: ' + xhr.status);
      }
    }
  };

  // Send request
  xhr.send();
}

// Use the handwritten AJAX to send a GET
ajax('https://api.example.com/data', 'GET', function(response) {
  console.log(response);
});
```

### Familiar with message queues / middleware?

Message middleware is a software system used in distributed systems to enable async communication between components. It acts as a broker that delivers messages between producers (senders) and consumers (receivers). Commonly used for decoupling system components, improving scalability and reliability.

### Key features of message middleware:

1. **Async communication**: Supports async communication — the producer doesn't wait for the consumer to finish processing.

2. **Decoupling**: Producers and consumers don't need to talk directly, which decouples and makes the system more flexible and scalable.

3. **Reliability**: Message middleware typically provides persistence, transactions, and acknowledgments to ensure reliable delivery.

4. **Load balancing**: Can distribute messages evenly to multiple consumers, improving throughput.

5. **Message queue**: Typically includes a message queue for pending messages, with support for queuing and priority.

### Common message middleware products:

- **Apache Kafka**: A distributed streaming platform widely used for real-time data pipelines and streaming apps.

- **RabbitMQ**: An open-source message broker supporting many protocols, easy to use and scale.

- **ActiveMQ**: An open-source broker and integration server supporting many protocols and transports.

- **Redis**: Primarily an in-memory data store, but its pub/sub feature can also serve as message middleware.

### Use cases for message middleware:

- **Async processing**: Order processing, email sending, etc.

- **Decoupling**: Inter-service communication in microservices.

- **Traffic shaping**: Smoothing spikes — e.g. e-commerce flash sales.

- **Log collection**: Centralized log collection and processing.

- **Real-time data processing**: Real-time analytics, alerting, etc.

### Why can't we use symmetric encryption entirely?

Key exchange problem.

A main issue with symmetric encryption is key exchange. With symmetric encryption, both parties must share the same key. On the internet, safely sharing keys is challenging — keys can be intercepted in transit. If a third party intercepts the key, all communication can be decrypted.

### Initial trust problem

Symmetric encryption requires both parties to establish trust before communication to safely exchange keys. This is hard on the internet because users may have never met and have no other way to verify identity.

### 3. Key management problem

Another issue is key management. In large-scale network communication, every pair of users needs a unique key. As users grow, the number of keys grows multiplicatively, making management complex and difficult.

### 4. Security issues

Symmetric algorithms are typically faster than asymmetric ones but often less secure. Asymmetric algorithms (RSA, ECC) use a key pair (public/private), where the public key can be shared and the private key is kept secret. This makes key exchange safer — even if the public key is intercepted, communication can't be decrypted without the private key.

### 3. How to build cinema seat selection with Canvas on the frontend?

You can implement cinema seat selection with Canvas by drawing seats and handling click events.

```javascript
const canvas = document.getElementById('seat-map');
const ctx = canvas.getContext('2d');

const seats = [
    { x: 50, y: 50, width: 30, height: 30, status: 'available' },
    { x: 100, y: 50, width: 30, height: 30, status: 'occupied' },
];

function drawSeats() {
    seats.forEach(seat => {
        ctx.fillStyle = seat.status === 'available' ? 'green' : 'red';
        ctx.fillRect(seat.x, seat.y, seat.width, seat.height);
    });
}

canvas.addEventListener('click', (event) => {
    const rect = canvas.getBoundingClientRect();
    const x = event.clientX - rect.left;
    const y = event.clientY - rect.top;

    seats.forEach(seat => {
        if (x >= seat.x && x <= seat.x + seat.width && y >= seat.y && y <= seat.y + seat.height) {
            if (seat.status === 'available') {
                seat.status = 'selected';
                drawSeats();
            }
        }
    });
});

drawSeats();
```

### **Observer pattern vs Pub-Sub pattern**

| Aspect | Observer pattern | Pub-Sub pattern |
| :--- | :--- | :--- |
| **Communication** | Direct (subject notifies observers directly) | Indirect (through an event bus) |
| **Coupling** | Tightly coupled (subject and observer know each other) | Loosely coupled (publisher and subscriber decoupled) |
| **Roles** | Subject and Observer | Publisher, Subscriber, Event Bus |
| **Implementation complexity** | Simple | More complex (requires an event bus) |
| **Use cases** | Direct dependencies between objects | Decoupled scenarios, event-driven architecture |
| **Sync/async** | Usually synchronous | Can be either sync or async |
